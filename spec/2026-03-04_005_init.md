## 1) One-sentence pitch
Build a CLI-first research/report pipeline where Claude Code orchestrates source collection (YouTube and URLs), pushes curated sources into NotebookLM, requests analyses and deliverables (slides, infographic, briefing), and downloads artifacts into a local project folder with a reproducible evidence trail.

## 2) Problem and motivation
### What pain exists today?
- Research from videos and web sources is manual, slow, and hard to reproduce.
- “Agent research” often becomes low-quality web search with weak provenance.
- I want deliverables (slide decks, infographics, briefings) without burning model tokens or time.

### Why existing solutions are not enough
- Local RAG stacks are expensive to build and maintain.
- Manual NotebookLM workflows do not integrate cleanly into my CLI and repo workflow.
- Most automation fails on provenance, versioning, and repeatability.

## 3) Goals (must be true when we ship)
1. One command can: collect sources, curate, create/update a NotebookLM notebook, ask analysis questions, generate deliverables, download artifacts.
2. Every run produces an auditable run manifest: source list, filters, prompts, notebook id, artifact paths, timestamps.
3. Works reliably on macOS (MacBook Pro) with Python and bash, and integrates as Claude Code skills.

## 4) Non-goals (explicitly out of scope)
- Building a custom RAG backend or vector database.
- Scraping paywalled sites or bypassing platform access controls.
- A full GUI. CLI and NotebookLM UI are enough.

## 5) Users and use cases
### Primary user
- Me (developer, CLI-first, Python/bash/R)

### Secondary user
- Optional collaborators who can run the same pipeline and reproduce outputs (later phase)

### Top workflows (user stories)
- As Emmanuel, I want to query “topic X”, ingest 10 to 50 high-signal sources, and receive a slide deck + infographic grounded in those sources.
- As Emmanuel, I want to run the exact same pipeline later and verify what changed (sources, prompts, artifacts).
- As Emmanuel, I want to feed in my own PDFs/notes plus YouTube links, not only YouTube.

## 6) Success criteria
### Product metrics
- Setup time on a fresh Mac: <= 15 minutes.
- Time from query to downloaded deck: <= 10 minutes for 20 sources (excluding NotebookLM processing delays).
- Reproducibility: rerun produces identical manifest structure and stable filenames (content may differ if sources change).

### Technical metrics
- Reliability: 95% of runs succeed without manual intervention after auth is configured.
- Cost: no paid APIs required beyond my existing accounts.
- Maintainability: minimal dependencies, pinned versions, clear failure messages.

## 7) Constraints and assumptions
### Hard constraints
- Platforms: macOS.
- Languages: Python + bash as first-class, optional R for post-processing.
- NotebookLM source cap: 50 sources per notebook.
- Authentication: Google login must be interactive at least once; secrets must not be committed.

### Existing stack and boundaries
- Claude Code skills used as UX entrypoints.
- NotebookLM automation via an unofficial client library/CLI (treat as brittle).
- YouTube metadata via yt-dlp or equivalent.

## 8) Functional requirements (what it must do)
### Must-have
- [ ] CLI command: `run` that takes query + filters and produces artifacts locally.
- [ ] Source ingestion:
  - YouTube search (query, count, recency window, min views, max duration, exclude terms).
  - Direct URL list input (text file).
  - Local files (PDF, md, txt) optionally uploaded to NotebookLM where supported.
- [ ] Curation step before NotebookLM upload:
  - Deduplicate sources.
  - Enforce cap (default 20, max 50).
  - Emit `sources.json`.
- [ ] NotebookLM operations:
  - Create notebook or reuse an existing notebook id/name.
  - Add sources.
  - Ask one or more analysis prompts.
  - Generate deliverables: slide deck + infographic at minimum.
  - Download deliverables to `outputs/`.
- [ ] Provenance:
  - Write `outputs/run_manifest.json` including exact prompts and source metadata.
  - Save raw YouTube metadata JSONL used for selection.
- [ ] Claude Code skills:
  - `/yt-search` to generate curated sources list.
  - `/nlm` (or `/notebooklm`) to push sources, ask, generate, download.
  - A higher-level `/research` skill that chains both.

### Should-have
- [ ] `--dry-run` mode that stops after curation and prints what would be uploaded.
- [ ] `--resume` mode to continue after partial failure (auth, rate limits, interrupted download).
- [ ] Structured prompt templates stored in repo (not hard-coded in skills).

### Nice-to-have
- [ ] R helper to render a “methods + provenance” appendix slide from the manifest.
- [ ] Weekly scheduled run (outside scope for v1 unless trivial).

## 9) UX and UI notes
- UX surface is CLI + Claude Code slash commands.
- Output is a predictable folder tree:
  - `outputs/<run_id>/` containing artifacts + manifest
- Error messages must say exactly what to do next (login, reduce sources, retry).

## 10) Data model and state
### Core entities
- RunManifest:
  - run_id, timestamp, query, filters
  - sources[]: {type, url/path, title, channel, views, duration, publish_date, selected_reason}
  - notebook: {name, id}
  - prompts[]: {name, text}
  - artifacts[]: {type, filename, sha256, created_at}
  - status: success|partial|failed + error summary

### Data lifecycle
- Creation: new run creates new folder under outputs, always append-only.
- Updates: rerun creates a new run_id, never overwrites previous outputs.
- Deletion/retention: manual cleanup only (no auto-delete).
- Auditing/logging: logs go to `outputs/<run_id>/run.log`.

## 11) API and integration surface
### Public interfaces
- CLI:
  - `research run --query ...`
  - `research sources --query ...` (curate only)
  - `research notebook --name ... --add sources.json`
- Claude skills call these CLIs, not internal python functions.

### External integrations
- YouTube via yt-dlp: metadata search and extraction.
- NotebookLM via unofficial library/CLI: create/add/ask/generate/download.

## 12) Security, privacy, and abuse cases
- Threats:
  - Accidental commit of auth tokens/cookies.
  - Uploading sensitive local docs into NotebookLM unintentionally.
- Permissions model:
  - Single-user local tool in v1.
- Data privacy:
  - Default: only public URLs are uploaded.
  - Local file upload must be opt-in with explicit flag.
- Abuse prevention:
  - Rate-limit YouTube calls.
  - Cap number of sources and retries.

## 13) Edge cases and failure modes
- NotebookLM auth fails or expires: tool must detect and instruct to re-login.
- Source cap exceeded: tool must prune deterministically and report what was dropped.
- Video removed, captions missing, or region-locked: tool must skip and backfill.
- NotebookLM API changes break the client: tool must fail fast with clear guidance and a manual fallback flow.

## 14) Operational plan
- Observability:
  - structured logs + run manifest.
- Alerts:
  - none in v1.
- Backups:
  - outputs are local and versioned via git-ignore, optionally synced via your cloud.
- Feature flags:
  - simple config file toggles are enough.

## 15) Testing strategy
- Unit:

- Integration:

- E2E:
  - optional manual test script that runs on 3 public videos.

## 16) Rollout plan
- Phase 1: local CLI works end-to-end with one notebook and one deliverable.
- Phase 2: Claude Code skills wrap the CLI.
- Phase 3: add templates, resume mode, richer artifacts.
