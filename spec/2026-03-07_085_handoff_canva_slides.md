# Spec: Deliverables Handoff Guide
Date: 2026-03-07
Status: v1

---

## Goals

Write `outputs/<run_id>/deliverables_handoff.md` automatically at the end of a qualifying `nlm-orch run`. The file gives the user step-by-step instructions for making downloaded deliverables editable in Canva or Google Slides, a post-editing checklist, and provenance references.

## Non-goals

- No new CLI flags in v1.
- No third-party API calls.
- No changes to NotebookLM operations, curation pipeline, `sources.py`, `notebooklm_cli.py`, or `pyproject.toml`.
- Not added to `docs/outputs_template/` (conditional output; spec + manifest field are sufficient documentation).

---

## Trigger condition

Write `deliverables_handoff.md` if and only if **both** of the following are true:

1. At least one of `slides` or `infographic` is in `args.deliverables`.
2. At least one visual artifact was successfully downloaded (status `"downloaded"` in `artifacts_result` for keyword `slides` or `infographic`).

```python
visual_keywords = ("slides", "infographic")
has_keyword = any(kw in args.deliverables for kw in visual_keywords)
has_download = any(
    a["status"] == "downloaded"
    for a in artifacts_result
    if a["keyword"] in visual_keywords
)
should_write = has_keyword and has_download
```

If all visual downloads failed, do not write the file. The user has nothing to edit yet.

This check happens after deliverable downloads complete (step 10 in `cmd_run`).

Not written in `--dry-run` or `--review` mode (no deliverables generated in those modes).

---

## Output path

```
outputs/<run_id>/deliverables_handoff.md
```

## File contents

Generated by `_write_handoff(...)`. All content is static prose or derived from runtime parameters. No external files read, no network calls.

### Intro paragraph

One short paragraph before Section 1:

```markdown
Your deliverables are ready. Use the steps below to make them editable,
apply your branding, and export in the format your audience needs.
```

### Section 1: Generated deliverables

Heading: `## Generated deliverables`

A bullet list of each deliverable that was in `--deliverables` (all keywords, including `briefing` if requested). For each:
- The keyword and filename
- Status: `downloaded` (file exists) or `not downloaded` (failed or absent)
- Absolute path if downloaded, else `-`

Example:
```markdown
## Generated deliverables

- **slides** - `outputs/<run_id>/artifacts/deck.pdf` (downloaded)
- **infographic** - `outputs/<run_id>/artifacts/infographic.png` (downloaded)
- **briefing** - `outputs/<run_id>/artifacts/briefing.md` (downloaded)
```

Show only deliverables that were in `--deliverables`. No em dashes; use plain ` - ` separator.

### Section 2: Make editable in Canva

Heading: `## Make editable in Canva`

Always present when the file is written (at least one of slides/infographic triggered the file).

Show subsections only for keywords that were in `--deliverables`:

```markdown
## Make editable in Canva

### Slide deck (deck.pdf)

1. Go to canva.com and click **Create a design**.
2. Choose **Import** and upload `artifacts/deck.pdf`.
3. Canva will convert each page to an editable slide. Review layout fidelity.
4. Apply your brand colors and fonts using **Brand Kit** (Canva Pro) or manually per slide.
5. Replace placeholder images if any were inserted by NotebookLM.
6. Export as **PowerPoint (.pptx)** or **PDF** when done.

### Infographic (infographic.png)

1. Go to canva.com and click **Create a design -> Custom size**.
2. Import the PNG as a background image using **Uploads -> Upload files**.
3. Overlay editable text boxes and shapes on top to make content modifiable.
4. Alternatively, use **Edit photo -> Background Remover** to isolate elements (Canva Pro).
5. Export as PNG (for web) or PDF (for print).
```

URLs are plain text (no Markdown hyperlinks).

### Section 3: Make editable in Google Slides

Heading: `## Make editable in Google Slides`

Show this section when `slides` OR `infographic` is in `--deliverables` (i.e., always when the file is written).

Show the PDF subsection only when `slides` was requested. Show the PNG subsection only when `infographic` was requested.

```markdown
## Make editable in Google Slides

### Slide deck (deck.pdf)

1. Open Google Drive (drive.google.com) and click **New -> File upload**.
2. Upload `artifacts/deck.pdf`.
3. Right-click the uploaded PDF and choose **Open with -> Google Slides**.
4. Google will convert the PDF to an editable presentation. Text and layout fidelity varies.
5. Apply theme colors via **Slide -> Edit theme**.
6. Download as **PowerPoint (.pptx)** or share directly from Drive.

### Infographic (infographic.png)

1. Open Google Drive (drive.google.com) and click **New -> File upload**.
2. Upload `artifacts/infographic.png`.
3. Open Google Slides and create a new blank presentation.
4. Insert the PNG via **Insert -> Image -> Upload from computer**.
5. Resize to fill the slide. Overlay text boxes and shapes as needed.
6. Export via **File -> Download -> PNG image** or share directly from Drive.
```

URLs are plain text.

### Section 4: Post-editing checklist

Heading: `## Post-editing checklist`

Static, always present:

```markdown
## Post-editing checklist

- [ ] Apply brand colors and fonts to all slides/graphics.
- [ ] Verify citations: each claim should trace back to a source in `sources.json`.
- [ ] Review slide order and narrative flow against the original query.
- [ ] Remove or replace any NotebookLM watermarks or boilerplate footers, if permitted by your licence.
- [ ] Check image resolution before print export (infographic.png should be 300 DPI minimum for print).
- [ ] Export in the required format (pptx, pdf, png, svg) for your audience.
- [ ] Archive this run folder (`outputs/<run_id>/`) for reproducibility.
```

### Section 5: Provenance

Heading: `## Provenance`

Dynamic, always present:

```markdown
## Provenance

- **Run ID:** `<run_id>`
- **Query:** `<query>`
- **Manifest:** `outputs/<run_id>/run_manifest.json`
- **Sources:** `outputs/<run_id>/sources.json`

The manifest records all filters, prompt files used, NotebookLM notebook ID, and artifact download statuses.
```

## `_write_handoff` helper

New function in `cli.py`, alongside `_write_curation_report`:

```python
def _write_handoff(
    handoff_path: Path,
    deliverables: list[str],
    artifacts_result: list[dict],
    query: str,
    run_id: str,
    run_dir: Path,
) -> None:
    """Write deliverables_handoff.md. Called only when trigger condition is met."""
```

Parameters:
- `handoff_path`: destination file path (`run_dir / "deliverables_handoff.md"`)
- `deliverables`: `args.deliverables` list (e.g. `["slides", "infographic"]`)
- `artifacts_result`: list of artifact dicts from step 10, each with `keyword`, `filename`, `status`, optionally `path`
- `query`, `run_id`, `run_dir`: from the current run context

Reads no external files. Raises no exceptions (caller wraps in try/except).

---

## Call site in `cmd_run`

After step 10 (deliverable downloads), before saving the final manifest:

```python
visual_keywords = ("slides", "infographic")
has_keyword = any(kw in args.deliverables for kw in visual_keywords)
has_download = any(
    a["status"] == "downloaded"
    for a in artifacts_result
    if a["keyword"] in visual_keywords
)
handoff_path = None
if has_keyword and has_download and not args.dry_run and not in_review:
    _handoff = run_dir / "deliverables_handoff.md"
    try:
        _write_handoff(_handoff, args.deliverables, artifacts_result, args.query or "", run_id, run_dir)
        handoff_path = str(_handoff.resolve())
    except Exception as exc:
        logger.warning("deliverables_handoff.md write failed: %s", exc)
mstate["handoff_path"] = handoff_path
```

## Manifest change

Add `handoff_path` to `run_manifest.json`:

```json
"handoff_path": "outputs/<run_id>/deliverables_handoff.md"
```

Set to `null` when not written (briefing-only runs, all downloads failed, dry-run, review).

`mstate["handoff_path"]` initialized to `null` at the top of `cmd_run` (alongside other mstate fields).

## Exit code and error handling

Writing `deliverables_handoff.md` must never affect the run exit code. If the write fails, log a warning and continue. The file is informational; a write failure is non-fatal.

---

## Constraints and assumptions

- Artifact dicts in `artifacts_result` have at minimum: `keyword` (str), `filename` (str), `status` (str: `"downloaded"` or `"failed"`). Optionally `path` (str, absolute).
- The `path` field in the artifact dict is used for Section 1 display when status is `downloaded`. If absent, fall back to constructing `run_dir / "artifacts" / filename`.
- No external URLs as Markdown hyperlinks -- plain text only.
- No em dashes in generated content -- use ` - ` or `->` as separators.

---

## Acceptance criteria

1. File written when `slides` in deliverables AND slides download succeeded.
2. File written when `infographic` in deliverables AND infographic download succeeded.
3. File written when both `slides` and `infographic` in deliverables AND at least one downloaded.
4. File NOT written when `briefing` only in deliverables.
5. File NOT written when all visual downloads failed (both slides and infographic absent).
6. File NOT written in `--dry-run` mode.
7. File NOT written in `--review` mode.
8. Section 1 correctly shows `downloaded` for artifacts that exist, `not downloaded` for those that failed.
9. Google Slides section present when `slides` in deliverables (slides subsection shown).
10. Google Slides section present when `infographic` in deliverables (infographic PNG subsection shown).
11. Google Slides section present whenever file is written (one or both subsections).
12. Canva section always present when file is written.
13. Checklist section always present when file is written.
14. Provenance section includes run_id, query, and paths to manifest and sources.
15. Write failure is non-fatal (run exits 0 or with its normal code).
16. `handoff_path` in manifest set to resolved path when written, `null` otherwise.
17. Intro paragraph present before Section 1.
18. No Markdown hyperlinks in generated file (URLs are plain text).
